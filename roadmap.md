# Chrome2Code — Development Roadmap

## What Is This?

A development tool that captures browser errors, network failures, and user actions from a Chrome extension, and generates structured prompt files that developers feed to Claude Code to fix issues in their codebase.

## Architecture

```
Chrome Extension (capture) → JSON export → Project folder → CLI (generate prompts) → Developer runs with Claude Code
```

### Components

1. **Chrome Extension** — Captures console errors, failed network requests, user actions (clicks, navigation, inputs), and console logs. Exports a JSON incident file.
2. **CLI Tool (`chrome2code`)** — Reads incident JSON files from the project folder, generates development prompt files in a `prompts/` directory. Runs in the project root alongside CLAUDE.md.
3. **Claude Code** — Developer runs the generated prompts. Claude Code reads CLAUDE.md + project files and applies fixes. This is the developer's responsibility, not the tool's.

### Future (Post-MVP)

4. **Server API** — Remote incident storage, token-based auth, incident portal where developers can browse, triage, and mark incidents as solved.
5. **Incident Portal** — Web UI to review captured incidents, mark resolved, prioritize next fixes.

## How It Works

1. Developer tests their app in the browser
2. Extension records a rolling buffer of user actions (last 30 events)
3. When an error occurs, extension snapshots: actions, errors, network failures, console logs, route URL
4. Developer exports the incident as a JSON file
5. Developer drops the JSON into `.chrome2code/incidents/` in their project
6. Developer runs `chrome2code generate`
7. CLI reads the incident JSON files and generates `.prompt.md` files in `prompts/`
8. Developer runs `claude "$(cat prompts/incident-name.prompt.md)"` to fix the issue
9. Developer tests the fix with hot reload, moves to next incident

## Project Structure

```
~/my-project/
├── CLAUDE.md                          # Project context for Claude Code
├── src/
├── .chrome2code/
│   ├── config.json                    # CLI config (future: server URL, token)
│   └── incidents/
│       ├── login-crash.json           # Exported from extension
│       └── order-list-500.json
└── prompts/
    ├── login-crash.prompt.md          # Generated by CLI
    └── order-list-500.prompt.md
```

## Incident JSON Schema

```json
{
  "id": "login-crash",
  "url": "http://dev.myapp.com:3000/dashboard/orders",
  "timestamp": "2026-02-24T10:30:00Z",
  "actions": [
    { "type": "click", "target": "button#load-orders", "text": "Load Orders" },
    { "type": "input", "target": "input#search", "field": "search" },
    { "type": "navigate", "from": "/dashboard", "to": "/dashboard/orders" },
    { "type": "xhr", "method": "GET", "url": "/api/orders", "status": 500, "response": "{\"error\":\"connection refused\"}" },
    { "type": "error", "message": "TypeError: Cannot read property 'map' of undefined", "stack": "at OrderList.render (src/components/OrderList.jsx:24)" }
  ],
  "console_logs": [
    "Fetching orders for user 42",
    "API base URL: http://localhost:8080"
  ],
  "notes": "Happens after clicking load orders when backend is slow"
}
```

## Data Captured by Extension

| Category | What | How |
|---|---|---|
| User actions | Clicks, inputs, form submits, scrolls | Content script DOM event listeners |
| Navigation | Route changes, URL hash changes | pushState/popState listeners |
| Console errors | Runtime errors, unhandled rejections | window.onerror, onunhandledrejection |
| Network failures | 4xx/5xx responses, timeouts, CORS | chrome.devtools.network / webRequest API |
| Console logs | Developer-added console.log output | Console API override |
| Page context | Current URL, route, timestamp | Location API |

## Development Phases

### Phase 1 — MVP (No Server)

**Goal:** Validate that generated prompts lead to good fixes.

#### 1.1 Chrome Extension
- [ ] Manifest V3 setup
- [ ] Content script: capture clicks, inputs, navigation
- [ ] Background script: capture network requests (webRequest API)
- [ ] DevTools panel: capture console errors and logs
- [ ] Rolling action buffer (last 30 events)
- [ ] Error snapshot: freeze buffer on error
- [ ] Export incident as JSON (download or copy to clipboard)
- [ ] Simple popup UI: start/stop recording, view captured incidents, export

#### 1.2 CLI Tool (`chrome2code`)
- [ ] `chrome2code init` — create `.chrome2code/` and `prompts/` directories
- [ ] `chrome2code generate` — read all incident JSONs, generate prompt markdown files
- [ ] `chrome2code generate <name>` — generate prompt for a specific incident
- [ ] `chrome2code list` — show all incidents and their status (new/prompted/resolved)
- [ ] `chrome2code resolve <name>` — mark an incident as resolved
- [ ] Prompt template engine — structured markdown with error, actions, network, context
- [ ] Publish as npm package

### Phase 2 — Server + Portal

**Goal:** Multi-device support, team collaboration, incident management.

- [ ] Server API (Node.js + Express/Fastify)
- [ ] Auth: JWT token-based registration and login
- [ ] Database: PostgreSQL for incidents, users, projects
- [ ] REST endpoints: POST incidents, GET incidents, PATCH status
- [ ] Extension sends incidents directly to server (replaces JSON export)
- [ ] CLI pulls incidents from server (`chrome2code pull`)
- [ ] Web portal: browse incidents, triage, mark resolved
- [ ] Team support: shared project incidents

### Phase 3 — Enhancements

- [ ] Source map support — resolve minified stack traces to original file paths
- [ ] Auto-detect framework/stack from project files to enrich prompts
- [ ] Incident grouping — deduplicate similar errors
- [ ] Severity scoring — prioritize incidents
- [ ] Session replay — richer action capture with timestamps and DOM snapshots
- [ ] `chrome2code watch` — watch for new incident files, auto-generate prompts
- [ ] Integration with CI/CD — capture errors from automated testing

## Key Design Decisions

1. **Tool generates prompts, not fixes.** Claude Code handles the fixing. This keeps the tool code-agnostic and simple.
2. **User controls execution.** The developer decides which prompts to run and when. No auto-fixing.
3. **CLAUDE.md is for Claude Code, not for the CLI.** The CLI generates clean incident data. Claude Code reads CLAUDE.md when the developer runs the prompt.
4. **MVP is local-only.** No server until prompt quality is validated with real errors.
5. **Action replay is the differentiator.** The sequence of user actions before the error is what makes this more valuable than just copy-pasting an error message.
